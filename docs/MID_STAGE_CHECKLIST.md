# 中期验收清单

用于部署前和给学长/队友体验前的快速检查。

## 1. 登录 / 注册

- [ ] 顶部能看到当前状态：未登录或当前昵称。
- [ ] 点击用户入口会打开弹窗。
- [ ] 弹窗包含昵称、出行偏好、预算偏好。
- [ ] 保存后 profile 写入 localStorage。
- [ ] 退出后回到未登录访客状态。

## 2. 用户历史隔离

- [ ] 登录用户 A 生成路线后，右侧规划记录出现 A 的历史。
- [ ] 切换到用户 B 后，看不到用户 A 的历史。
- [ ] 退出登录后，只看到未登录访客历史。
- [ ] 重新登录用户 A 后，能看到用户 A 自己的历史。

## 3. Session 同步

- [ ] 点击历史记录后，左侧输入框和右侧路线一致。
- [ ] 当前场景/画像和路线说明一致。
- [ ] 用户偏好说明和当前 session 一致。
- [ ] 不出现“输入是 A，路线是 B”的错位。

## 4. 偏好影响路线

- [ ] 选择“安静”后，路线理由或候选更偏安静地点。
- [ ] 选择“省钱”或填写预算偏好后，预算约束进入“已理解”和路线排序。
- [ ] 选择“少排队”后，高排队 POI 不应被优先推荐。
- [ ] 选择“亲子友好”后，成人夜生活 POI 不应优先出现。

## 5. 路线可信度

- [ ] 至少 3 个 POI 串联。
- [ ] 预算展示清楚：预算内、略超、超预算。
- [ ] 营业时间展示清楚，到店时是否营业可判断。
- [ ] 排队风险展示清楚，不只是一个分数。
- [ ] 区域不明显跳脱；如果跳脱，需要路线解释能说得通。
- [ ] 首页不显示“步行 0 min”；车程为主时显示车程或移动成本。

## 6. Replan

- [ ] “换一家评分更高的餐厅”只替换餐饮节点。
- [ ] “预算降到 300”优先替换超预算节点。
- [ ] “不要太赶”减少移动或节点压力。
- [ ] 修改后路线重新校验预算、营业、排队、移动成本。
- [ ] 提示语说明改了哪里，且其他节点尽量保留。

## 7. 首页信息架构

- [ ] 首页第一眼是路线产品，不是算法调试台。
- [ ] 用户首页不出现 `score`、权重、惩罚、mock map leg 等算法黑话。
- [ ] 用户可见文案以“预算内、排队低、少步行、车程、适合接电话”等人话表达。
- [ ] 技术证据保留在“查看规划依据 / 评委附录”。

## 8. 评委附录

- [ ] 明确展示链路：parseConstraints -> retrieveCandidates -> scorePOIs -> buildRouteCandidates -> validateRoute -> repair/replan -> explainRoute。
- [ ] 每一步有一句中文解释，说明抽取、召回、推荐、组合、校验、修复、解释做了什么。
- [ ] 有候选 POI 排序和推荐理由。
- [ ] 有约束校验与修复记录。
- [ ] 有数据源说明：当前 mock 数据源对齐真实接口，未来可替换高德和美团/点评数据。

## 9. 部署前命令

```bash
npm run build
npm run eval
```

如果 `npm run eval` 在沙盒环境因 `tsx` IPC 权限失败，使用：

```bash
node --import tsx scripts/runEval.ts
```

## 10. Vercel 配置

- [ ] Build Command: `npm run build`
- [ ] Output Directory: `dist`
- [ ] Install Command: `npm install`
- [ ] 不配置真实 API key。
- [ ] 部署后重新检查登录、历史、replan 和评委附录。
