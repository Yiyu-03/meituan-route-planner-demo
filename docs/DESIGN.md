# 漫游·手帐 — Agent Loop 与产品化思维设计

> 这份文档讲两件事:**这个 Agent 是怎么设计的**,以及**产品思维如何贯穿每一个技术决策**。
> 在线体验:<https://meituan-route-planner-demo.vercel.app>

---

## 0. 一句话立场:路线是"对象",不是"文本"

最容易的做法是让大模型直接吐一段攻略文本。但那东西**不可执行、会幻觉地点、无法校验预算与营业时间、同一句话每次结果都不一样**。

我们的立场:**路线是一个结构化对象**——经过需求理解 → 真实候选召回 → 个性化打分 → 路线组合 → 体检 → 修复 → 排序 → 真实路径接腿,最后才由大模型写"推荐理由"。大模型只点缀两端,不负责"拍脑袋造路线"。

这一条立场,决定了下面所有设计。

## 1. Agent Loop 设计

### 1.1 双层结构:ReAct 外壳 + 确定性骨架

```
┌─ ReAct 外壳(对话式,负责"想清楚要什么")──────────────┐
│   reason ──▶  searchPOI │ askUser │ finish              │
│   (定位城市 → 解析需求 → 决定搜什么 → 够了就收尾)        │
└───────────────────────┬─────────────────────────────────┘
                        │ finish / 线性回退
┌───────────────────────▼─ 确定性骨架(负责"拼出可行路线")─┐
│  understand → retrieve → score → build(beam search)     │
│           → validate → repair → rank → attachLegs        │
│           → explain                                      │
└──────────────────────────────────────────────────────────┘
```

- **ReAct 外壳**(`lib/agent/react.ts`):像人一样"推理→行动→观察"。它决定**搜什么关键词、什么时候信息够了、要不要反问用户**。每一步只产出一个严格 JSON,思考被流式吐给前端。
- **确定性骨架**(`lib/agent/loop.ts` 等):拿到真实候选后,用打分 + beam search 组合路线,做营业/预算/时间/类目的体检,必要时只替换冲突节点(repair),最后接真实步行/车程腿。

### 1.2 为什么是"混合",而不是纯 LLM 或纯规则

| | 纯 LLM | 纯规则 | 混合(本项目) |
|---|---|---|---|
| 语义理解 | 强 | 弱 | LLM 做 |
| 约束满足/最优化 | 不可靠、会算错 | 强 | 确定性代码做 |
| 可复现 | 否(每次不同) | 是 | 是 |
| 泛化(新城市/新诉求) | 强 | 差(硬规则写死) | 强(理解交给 LLM) |
| 会不会编造地点 | 会 | 不会 | 不会(只用真实召回) |

让 LLM 做它擅长的(理解、文案),让确定性代码做它擅长的(满足约束、校验、修复)。**"退化/兜底这块让 LLM 做,而不是全靠泛化硬规则"** 是反复打磨出来的原则。

### 1.3 几个关键设计决策

- **绝不编造 POI**:候选只来自高德真实 `searchPOI`。数据不足 → 返回明确错误,而不是伪造一条外地路线。
- **思考可见即可信**:`thought / action / observation` 事件实时流式呈现——用户看得到"它在成都搜火锅、命中 24 家、选了更便宜的那家",黑盒变白盒。
- **askUser 是最后手段**:缺区域/街道/店名**不是**反问理由(这些能搜)。只有"用户到底想要什么"这一层缺失时才问。默认"搜"而非"问"。
- **配额与速度意识**:POI/路径结果进缓存;第一步把所有关键词一次性**并行**搜齐;尽量第 2 步就 finish。
- **适配推理模型**:DeepSeek 是 reasoning 模型(reasoning_content 吃 token),`max_tokens` 要给够、`timeout` 要放宽,否则 JSON 被截断会退化成"找不到亲子餐厅"。

### 1.4 refine(基于此方案修改)的设计

改方案不是"重新规划",而是**带上下文的最小改动**:

- **完整上下文喂给 LLM**:初始 query + 当前路线每一站(含人均/评分)+ 这次的修改诉求。
- **最小改动**:只动目标站,其余原样保留。
- **子品类保真**:"换更便宜的火锅"必须仍是火锅,而不是最便宜的随便一家餐厅(`cuisineOf` 只认店名,避免高德 tag 污染)。
- **尊重 criterion**:不让 LLM 把明确的"更便宜/更近/评分更高"降级成普通换一家;repair 换店时也按此意图排序,不撤销用户诉求。
- **反思事件防矛盾**:在路线最终确定后,基于**真实最终路线**再吐一条思考("第 1 站已换成 X,¥81→¥50,更省"),杜绝"改方案说换 A、路线却是 B"。
- **独立思考流模态**:refine 也有思考流,前端用"朱砂红批注"风格区分于首次规划。

## 2. 产品化思维设计

技术决策不是为了炫技,每一个背后都有产品判断。

### 2.1 前后端解耦 + 冻结契约 —— 让系统可演进

> "前端要什么 agent loop?把前后端解耦开,agent loop 只在后端,前端只负责显示。"

- **契约层 `contract/`**:SSE 事件的 zod schema + 类型,是前后端**唯一接缝**。
- 收益:前后端可并行开发、各自独立演进、契约可测试;前端永远是"纯展示",不藏业务逻辑。

### 2.2 把"思考"做成产品 —— 信任

规划是黑盒会让人不安。我们把 ReAct 的推理**流式呈现**,让用户看到 agent 在"怎么想":定位城市 → 读懂需求 → 搜索 → 观察命中 → 收尾。这不是 debug 日志,这是**产品的信任界面**。refine 时换成"正在改方案"的朱砂批注模态,强化"它在改你这一张"。

### 2.3 真实可用,而不是 demo mock —— 诚实

> "不做 mock,做可以直接小规模生产使用的产品。"

全面去 mock:真实高德 POI/路径、真实 DeepSeek、真实 Neon 数据库。**生产即 staging**。数据不足时**诚实兜底**(明确提示),而不是塞假数据骗用户。一个会编造地点的规划器,用户用一次就不信了。

### 2.4 社交传播:手帐分享卡 —— 让一次规划"可被晒出去"

把"一次路线规划"变成一张**可分享的手帐**:各站拍立得照片用朱砂"毛线针"缝在一起,缺图自动落 SVG 占位,一键导出 PNG / 调系统分享。这是产品的**自传播**钩子。

工程上,正是这些"产品细节"决定成败:
- 跨域照片 → 自建 `api/img-proxy` 同源代理,避免导出时画布被污染;
- 无 CORS 字体 CDN → 导出 `skipFonts` 降级,屏幕预览仍是文楷;
- 桌面 `share()` 失败 → 回退下载,而不是报"生成失败"。

### 2.5 体验细节里的产品判断

- **登录与服务分离**(`/login` 独立于 `/app`):进入服务前有"翻开手帐第一页"的仪式感。
- **多会话便签墙**:历史方案按账户/访客分别保存,随时翻回再改——规划是个反复的过程。
- **点击商铺地图定位**:即使地图漂移也能快速回到标定点。
- **真实步行/车程腿 + 移动构成**:不拿直线距离糊弄,腿按真实 mode 显示步行/车程。

### 2.6 设计语言:漫游·手帐 —— 质感即品牌

暖纸 + 墨黑 + 朱砂红;文楷(标题)/ Fraunces(拉丁数字)/ Noto Sans(正文);横格纸纹、颗粒噪点、和纸胶带、拍立得、印章。**不用通用 emoji 和 generic 图标**——连品牌印记都是自绘的 `RoamSeal`(印章里一条点状路线 + 目的地环)。质感本身就是与"又一个 AI 工具"的区隔。

---

## 3. 可复现性与质量保证

- **确定性骨架**让"同输入同画像 → 同结果",可单测、可评测。
- **真机并行 QA**:多城市 × 多人群并发跑(北京/上海/西安/广州/重庆/杭州…),逐条审路线合理性与文案。
- **QA 驱动的修复**(都是真机暴露出来的真问题):
  - 类目意图被吃掉(文化为主却出 3 餐厅)→ beam 加同类目超额惩罚;
  - 文案幻觉(凭空说"换成外滩")→ explain 硬约束只讲真实存在的站点;
  - 换火锅却给豌杂面 → 子品类保真(店名识别,排除 tag 污染);
  - 充电站混入景点 → 非目的地黑名单补"快电/充电桩"。

---

## 4. 还能往哪走

- 接入更多真实生活数据:排队、团购、UGC、人均的实时性;
- 预算从"软提示"升级为"硬约束 + 降档";
- 模糊亲子诉求下 ReAct 关键词更稳;同名 POI 去重;
- 把 refine 的"更近"也接入 repair 的意图排序。

  AgentThinking.tsx  思考流(plan / refine 两种模态)
